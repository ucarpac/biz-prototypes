①2025年期に試みたこと

  • 60日以内の案件は送客対象として、後段へ進める対象を広げた。
  • お客様対応でスピード優先の場合は、出張査定を待たせずセルフ査定をご案内して出品を上げる運用にした。
  • 埼玉・大阪エリアなどで出張査定員が離脱した分は、セルフ査定へ寄せて査定・出品の処理量を維持した。
  • オークションで流れた案件を、できるだけ交渉へ持ってくる運用にした。
  • 流れた案件は買取も積極的に行い、BID以外の成約機会を拾うようにした。
  • 流れた後の案件交渉は担当者の役割を変え、交渉・買取へつなげる体制にした。

④課題

  • 工数が逼迫しており、流れ案件の交渉・買取対応を継続拡大するには処理余力が課題。
  • 出品案内の効率化が必要。送客対象を広げた分、案内・確認・後追いの標準化が重要。
  • 人員の補充が必要。成果が出ている運用を維持するには、担当者の役割分担と出張査定員の離脱エリアを前提にした体制強化が必要。

確認結果として、通話可能/通話成功だけでなく、送客・査定・出品・売却も `master` のステータスscoreで管理されています。 ただし後段は「最新ステータス」ではなく「各scoreに初めて到達した日」で見る必要があります。 通話成功の前年差 -12,022 は、2025年期に `not_assessment_end` (score2) が大きく増え、2024年期に多かった `not_assessment_refuse` / `not_assessment` などのscore3ステータスが減った影響が大きいです。 そのため、実際の電話接続数そのものの前年差ではなく、master上のステータス分類差として解釈します。

集計方法

確認した `新CS!B2:B6` は `実施日基準(営業) / 全体 / 全体 / 全体 / 全体` です。`マージ` の参照列は、申込=P列、送客=Q列、査定=R列、出品=S列、成約=T列です。`実施日基準(経常)` との違いは成約基準で、経常はU列（車両引渡日）を参照します。このHTMLでは営業基準の成約日を使い、査定・出品の表示だけは期またぎによる逆転を避けるため、同じ期内で送客→査定→出品まで進んだ件数に補正しています。

`master` シートの定義では、案件ステップは 有効申込=1、通話可能=2、通話成功=3、送客=4、査定=5、出品=6、売却=7 です。

出張査定員とセルフ査定の確認元レポートでは、`ucarpac-uapp.gas_data.all_deals_view` を使い、基準日は `assessment_apply_datetime`、セルフ査定は `agent_shop = 【UcarPAC】（サポート）`、出張査定はそれ以外の有効な査定員として集計しています。期間Aは2024-07〜2025-06、期間Bは2025-07〜2026-04です。

  • 申込: `all_lp_view.application_date` で集計。
  • S有効申込: `all_lp_view.category = '送客'` を使用。
  • 通話可能: `guest_transitions.to_state` を `state_scores_master.score` に突合し、スコア2以上で集計。
  • 通話成功: `guest_transitions.to_state` を `state_scores_master.score` に突合し、スコア3以上で集計。
  • 送客: `all_deals_view.assessment_apply_datetime` で集計。
  • 査定: 同じ期内に送客され、同じ期内に査定された案件で集計。
  • 出品: 同じ期内に送客・査定され、同じ期内に出品された案件で集計。
  • 成約(BID): `all_deals_view.status = '成約'` かつ成約日基準。
  • 買取: `all_deals_view.status = '買取移行'` かつ成約日基準をproxyとして使用。
  • 売上: `手数料 + 車両価格 * 0.1 / 1.1 - 買取加算額 / 1.1 - BIDクーポン / 1.1`。
  • 台単価: 売上 ÷ 成約(BID)。
