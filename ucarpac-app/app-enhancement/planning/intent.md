# 真の意図の深掘り

## 1. この人は何の意思決定をしようとしているか？

UcarPACアプリに**どの追加機能を実装するか**を決めようとしている。
単なるアイデア出しではなく、最終的にHTMLプロトタイプまで作る前提なので、
「実装する機能の優先順位付きリスト」が意思決定の対象。

## 2. どんな結論が出れば「これで決められる」と言えるか？

- 他社事例に基づく「効果が実証済みの機能パターン」のリスト
- UcarPACの既存機能・ユーザー層との適合度評価
- 実装難易度と期待効果のマトリクス
- 優先度Top 3-5の機能が明確に特定されている
- 各機能の具体的なUI/UXイメージ（プロトタイプで表現可能なレベル）

## 3. 見落としていそうな論点

- **そもそも削除の主因は「機能不足」なのか？**
  - 通知がうるさい、UXが悪い、初期設定が面倒、等の可能性
  - 「価格が下がっている現実を見たくない」心理的離脱
- **ターゲットユーザーの利用シーン**
  - 車の売却検討者（短期利用が前提）vs 車の保有者（長期利用の余地あり）
  - 売却完了後にアプリを消すのは自然な行動では？
- **競合の定着率はどうなのか？**
  - 車関連アプリの業界平均削除率との比較
- **セルフ査定機能との相乗効果**
  - 追加機能がセルフ査定への導線を阻害しないか

## 4. 間違った結論を出した場合の最悪シナリオ

- 開発リソースを投じた機能が使われず、削除率が改善しない
- 既存の「価格モニタリング」というコアバリューがぼやける
- セルフ査定機能との優先度の食い合い
