# 機能比較マトリクス

## 評価基準

- **期待効果**: リテンション率改善への寄与度
- **実装難易度**: 開発工数・技術的複雑性（★多い=容易）
- **コアバリュー整合性**: 「車の価格トラッキング→売却」のコアバリューとの一致度
- **データロック効果**: ユーザーのスイッチングコストを生む度合い
- **競合差別化**: 他社が容易に模倣できないか

### 評価凡例
- ★★★: 要件を十分に満たす
- ★★☆: 条件付きで満たす
- ★☆☆: 不足あり

## トレードオフ要約

- **A（価格アラート強化）** はコアバリュー直結・低コストだが、Carvana/cars AIとの差別化が課題
- **E（オンボーディング改善）** は最もROIが高い可能性があるが、「開く理由」がなければ効果限定的
- **B（メンテリマインダー）** は実績豊富だが、オートバックス等の専用アプリと競合し、コアバリューとの距離がある
- **C（売却タイミングAI）** は差別化最大だが、実装コスト最大・効果が不確実
- **D（維持費管理）** はデータロック効果は高いが、UcarPACユーザーのニーズとの適合性が最も低い

## 推奨（条件付き）

### Phase 1（即時着手）: A + E の組み合わせ
- **条件**: 既存価格データが活用可能、開発リソース2-3人月以内
- **理由**: コアバリュー強化 + 初期体験改善のセットが最もROI高い
- 価格アラートで「定期的に開く理由」を作り、オンボーディングで初回離脱を減らす

### Phase 2（検証後）: B のシンプル版
- **条件**: Phase 1で削除タイミング分布が判明し、「保有期間中の利用価値」が課題と確認された場合
- **理由**: 車検リマインダーのみ（フル機能メンテ管理ではなく）なら低コストで実装可能

### Phase 3（差別化投資）: C
- **条件**: Phase 1-2で定着率改善が確認され、開発リソースに余裕がある場合
- **理由**: 長期的な競争優位だが、短期ROIは低い

## 逆転条件

| 条件変化 | 結論の変化 |
|---------|-----------|
| UcarPACの削除がDay 1に集中している場合 | → **Eが最優先に逆転**。機能追加よりオンボーディングが先 |
| ユーザーの80%以上が「売却意思なし・相場確認のみ」の場合 | → **Bが最優先に逆転**。売却以外の価値提供が必須 |
| UcarPACのオークションデータ精度が高い場合 | → **Cが最優先に逆転**。独自データで圧倒的差別化 |
| 開発リソースが1人月以下の場合 | → **Eのみ実施**。機能追加は見送り |
| 「ほぼすべての車査定アプリがサービス終了」が現在も継続している場合 | → **リポジショニングが最優先**。機能追加の前にアプリのコンセプト変更が必要 |

## 感度分析

### 仮定1: 「プッシュ通知の効果は車アプリにも適用できる」
- **現在の結論への影響**: 機能AとBの期待効果の根拠
- **崩れるケース**: 車アプリの通知は「不安を煽る」と受け取られ、逆効果になる可能性
- **崩れた場合**: A・Bの期待効果が★★☆に低下。Eの相対優位が拡大

### 仮定2: 「UcarPACユーザーは車の保有期間中もアプリを使う動機がある」
- **現在の結論への影響**: 全機能の前提
- **崩れるケース**: ユーザーの大半が売却完了後に即削除しており、保有期間中はそもそもアプリを開かない
- **崩れた場合**: 全機能の期待効果が大幅低下。「削除率改善」自体が誤った目標になる

### 仮定3: 「既存の価格データが十分な精度を持つ」
- **現在の結論への影響**: A・Cの実現可能性
- **崩れるケース**: オークション価格データのカバレッジが限定的（特定車種のみ等）
- **崩れた場合**: A・Cの実装が困難に。Bが最優先に逆転
